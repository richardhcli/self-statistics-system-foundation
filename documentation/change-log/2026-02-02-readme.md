## Journal Entry Schema Refactor: User-Centric Data Model

**Date**: 2026-02-02  
**Type**: Breaking Change - Schema Redesign  
**Impact**: Complete journal entry structure overhaul

### Summary
Transformed journal entry data model from system-centric (generation method) to user-centric (viewing perspective). Unified action weighting for both AI and manual modes, separated performance results from metadata, and introduced explicit flags for entry characteristics. This refactor eliminates legacy fields, simplifies component logic, and establishes a consistent schema across the entire journal system.

### New Schema Structure
```typescript
{
  content: string;                    // Raw entry text
  actions: Record<string, number>;    // Unified: action ‚Üí weight
  result?: {                          // Performance metrics
    levelsGained: number;
    totalExpIncrease: number;
    nodeIncreases: Record<string, number>;
  };
  metadata: {                         // Entry metadata
    flags: { aiAnalyzed: boolean };
    timePosted: string;
    duration?: string;
  };
}
```

### Key Changes

#### 1. Unified Action Weighting
- **Before**: Separate `weightedActions: WeightedAction[]` and `actions: string[]`
- **After**: Single `actions: Record<string, number>` for both modes
  - AI mode: Semantic weights (e.g., `{ "Debugging": 0.7, "Code review": 0.3 }`)
  - Manual mode: Default weights (e.g., `{ "Exercise": 1, "Reading": 1 }`)

#### 2. Separated Performance Results
- **Before**: Mixed in `metadata: { totalExp, levelsGained, nodeIncreases }`
- **After**: Dedicated `result` object for calculated metrics
- **Rationale**: Clear distinction between entry metadata and performance outcomes

#### 3. Explicit Flags System
- **Before**: Implicit determination via field presence
- **After**: `metadata.flags.aiAnalyzed` boolean
- **Benefit**: Explicit state tracking, extensible for future flags

#### 4. Legacy Fields Removed
- ‚ùå `weightedActions?: WeightedAction[]` (replaced by `actions` Record)
- ‚ùå `action?: string` (deprecated singular field)
- ‚ùå Top-level `duration` (moved to `metadata.duration`)
- ‚ùå `metadata.totalExp` (renamed to `result.totalExpIncrease`)

### Files Modified (12 total)

**Type Definitions:**
- `features/journal/types/index.ts` - Complete schema redesign

**Core Pipeline:**
- `features/journal/api/create-entry.ts` - Updated loading placeholder

### User Experience

**Auto-Submit Flow (Record button):**
1. User clicks Record ‚Üí speaks ‚Üí clicks Stop
2. Gemini transcribes WebM audio
3. Immediate entry creation with AI processing
4. No manual review required

**Manual Review Flow ("To Text" button):**
1. User clicks Record ‚Üí speaks ‚Üí clicks "To Text" (while recording)

- Audio format: `audio/webm` (MediaRecorder default)
- Max recording: 60 seconds (auto-stops)
- Timeout: 30 seconds for Gemini transcription

## Voice Recording: Progressive Entry Creation with MediaRecorder Fix

**Date**: 2026-02-02 (Late Evening)  
**Type**: Bug Fix + UX Enhancement  
**Impact**: Immediate user feedback + fixed transcription bug

### Summary
Fixed critical MediaRecorder data collection bug where audio wasn't properly captured before Blob creation (race condition). Implemented progressive entry creation for better UX: dummy entry appears immediately, then updates with transcribed text, then full AI analysis.

### Key Changes

**FIXED: MediaRecorder Data Collection Bug**
- **Problem**: `MediaRecorder.stop()` is asynchronous - `ondataavailable` fires after `stop()` returns
- **Previous code**: Called `stop()` then immediately created Blob (race condition = empty audio)
- **Solution**: Added Promise-based `stopRecordingAndCollectAudio()` that waits for stop event
- **Result**: All audio chunks properly collected before Blob creation

**NEW: Progressive Entry Creation**
- **Stage 1 (immediate)**: Dummy entry created with "üé§ Transcribing..." placeholder
- **Stage 2 (2-3s later)**: Entry content updated with transcribed text
- **Stage 3 (3-5s later)**: Entry fully populated with AI analysis (actions/skills/characteristics)
- **UX Benefit**: Users see immediate feedback instead of waiting 5-10 seconds for complete processing

**Before (Blocking):**
```
User stops recording ‚Üí [5-10 second wait] ‚Üí Entry appears with full data
```

**After (Progressive):**
```
User stops recording ‚Üí Entry appears (üé§ Transcribing...)
                    ‚Üí Entry updates (transcribed text)
                    ‚Üí Entry updates (full AI data)
```

### Technical Implementation

**Modified Files:**
- `voice-recorder.tsx`:
  * `stopRecordingAndCollectAudio()` - Promise-based audio collection with stop event
  * `createDummyEntry()` - Immediate placeholder with timestamp
  * `transcribeAndUpdateEntry()` - Background transcription + AI trigger
  * `cleanupRecordingResources()` - Separated from data collection
  * Removed race condition in audio blob creation

- `journal-feature.tsx`:
  * Progressive callback pattern instead of single text callback
  * `updateEntryWithTranscription()` - Updates entry content, triggers AI
  * Passes `journalActions` to voice recorder for dummy entry creation

- `types/index.ts`:
  * Updated `VoiceRecorderProps` to support progressive callbacks
  * Added `onUpdateEntryWithTranscription` callback
  * Added `journalActions` prop

### Error Handling
- Empty audio: Alert + no dummy entry created
- Transcription fails: Entry updates to "‚ùå Transcription failed"
- Transcription error: Entry updates to "‚ùå Transcription error" + alert
